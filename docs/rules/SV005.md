# "attempt to store a secret value as a table key"

**Rule SV005 — secretable API result used as a table key.** Severity: `error` (unconditionally
secret API), `warning` (conditionally secret, or via data-flow).

## The error this rule prevents

Seen in BugSack / the default error frame on WoW 12.0+ ("Midnight") when a
[secret value](https://warcraft.wiki.gg/wiki/Secret_Values) is used to index into or key a
table (wording varies with the operation):

```
attempt to store a secret value as a table key
table index is a secret value
```

Storing a secret as a table **value** is fine; using it as a **key** is not — key lookup would
reveal equality, which is exactly what secrets forbid. The star case is `UnitGUID`:
`seen[UnitGUID(unit)] = true` is one of the most common patterns in pre-12.0 addons and it
throws the moment the GUID arrives secret.

## Example that breaks in-game

```lua
local seen = {}
seen[UnitGUID("target")] = true            -- SV005
local cache = { [UnitName("focus")] = 1 }  -- SV005
```

## How to fix

Guard and re-key with non-secret data, or track existence differently:

```lua
local issecretvalue = issecretvalue or function() return false end

local guid = UnitGUID("target")
if not issecretvalue(guid) then
  seen[guid] = true
else
  -- re-key with data you are allowed to inspect, e.g. a counter or spellId+timestamp,
  -- or skip the cache for this unit
end
```

Player-controlled unit tokens are never identity-restricted, so these are **not** flagged:

```lua
seen[UnitGUID("player")] = true   -- not flagged
seen[UnitName("raid1")]  = 1      -- not flagged
```
