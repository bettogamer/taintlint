# "execution tainted by '*** ForceTaint_Strong ***'"

**Rule SV009 — secretable APIs inside `loadstring` / `RunScript` code.** Severity: `warning`.

## The error this rule prevents

Seen in BugSack / the default error frame on WoW 12.0+ ("Midnight"):

```
attempt to compare a secret number value (execution tainted by '*** ForceTaint_Strong ***')
attempt to perform arithmetic on a secret value (execution tainted by '*** ForceTaint_Strong ***')
```

Code compiled at runtime — `loadstring`/`load` string literals, `RunScript`, WeakAuras-style
custom trigger snippets — executes under **ForceTaint_Strong**. In that context every
secretable API (`UnitHealth`, `UnitHealthMax`, `UnitPower`, …) returns a secret value
**always, even out of combat**. There is no "test it out of combat" escape hatch: the snippet
that works in normal addon code throws the moment it runs compiled.

## Example that breaks in-game

```lua
local trigger = loadstring(
  "return UnitHealth('player') / UnitHealthMax('player') < 0.3")  -- SV009 ×2
```

## How to fix

Inside compiled code you cannot read combat numbers by arithmetic at all. Options, in order of
preference:

1. **Move the logic out of the compiled chunk** into normal addon code, guarded with
   `issecretvalue` as usual.
2. **Use secret-accepting APIs** inside the chunk: `StatusBar:SetValue`,
   `FontString:SetText`/`string.format`, `ColorCurve`, `Duration` — they consume secrets
   without your code inspecting them.
3. **Wrap in `pcall` + `issecretvalue` and degrade** — if you host user-provided code
   (WeakAuras-style), expose `issecretvalue` to the sandbox (with a pre-12.0 shim) and catch
   secret errors centrally so a bad snippet degrades instead of popping Lua errors.

## Notes

- taintlint only scans **string literals** passed to `loadstring`/`load`/`RunScript`.
  Dynamically assembled strings (user code from SavedVariables) are out of scope — a
  documented false negative; use the runtime degradation pattern (option 3) for those.
