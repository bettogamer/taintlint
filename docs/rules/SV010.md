# Returning a raw secret value propagates the hazard

**Rule SV010 — helper returning a raw secretable value.** Severity: `info`.

## The problem this rule prevents

No error fires on the `return` line itself — that is what makes this pattern dangerous. A
helper that returns the raw result of a secretable API hands a live grenade to **every
caller**: each of them must now guard the value, and the eventual explosion
(`attempt to perform arithmetic on a secret value`, `table index is a secret value`, …) is
reported far from its cause.

## Example

```lua
local function guidOf(unit)
  return UnitGUID(unit)   -- SV010: every caller inherits the secret
end

-- 500 lines later, in another file:
seen[guidOf("target")] = true   -- throws here, cause is up there
```

## How to fix

Answer the caller's actual question with a non-secret value:

```lua
local function unitExists(unit)
  local guid = UnitGUID(unit)
  if guid then
    return true
  end
  return false
end
```

(Boolean-testing a non-boolean secret is allowed, so `if guid then` is safe even when `guid`
is secret.)

If callers genuinely need the value, document the contract and guard at the single chokepoint
inside the helper instead of in N call sites.
