# "attempt to compare a secret number value"

**Rule SV002 — comparison of a secretable API result.** Severity: `error` (unconditionally
secret API), `warning` (conditionally secret, or via data-flow).

## The error this rule prevents

Seen in BugSack / the default error frame on WoW 12.0+ ("Midnight"):

```
attempt to compare a secret number value (execution tainted by 'MyAddon')
attempt to compare a secret number value (execution tainted by '*** ForceTaint_Strong ***')
```

On tainted execution paths, comparing a [secret value](https://warcraft.wiki.gg/wiki/Secret_Values)
with `== ~= < > <= >=` throws immediately. This is the classic "my addon only errors in combat"
bug: out of combat the API returns a plain number, in combat it returns a secret.

If the error says `*** ForceTaint_Strong ***`, the comparison happened inside dynamically
compiled code (`loadstring`/`RunScript`) — see [SV009](SV009.md): there, secretable APIs return
secrets **always**, even out of combat.

## Example that breaks in-game

```lua
if UnitHealth("target") > 0 then print("alive") end   -- SV002
local dead = UnitHealth("target") == 0                -- SV002
```

## How to fix

Guard before comparing, with a sensible fallback:

```lua
local issecretvalue = issecretvalue or function() return false end

local hp = UnitHealth("target")
if issecretvalue(hp) then
  -- degrade: hide the element, show "??", or skip the check
else
  if hp > 0 then print("alive") end
end
```

Still allowed on secrets, per Blizzard's rules:

- `type(v)` — works on secret values
- Boolean test of a **non-boolean** secret (`if UnitName("target") then … end`) is allowed
- Threshold-driven visuals: use `ColorCurve` / `StatusBar` instead of comparing numbers yourself
