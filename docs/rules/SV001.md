# "attempt to perform arithmetic on a secret value"

**Rule SV001 — arithmetic on a secretable API result.** Severity: `error` (unconditionally
secret API), `warning` (conditionally secret, or via data-flow).

## The error this rule prevents

Seen in BugSack / the default error frame on WoW 12.0+ ("Midnight"):

```
attempt to perform arithmetic on a secret value
attempt to perform arithmetic on local 'hp' (a secret number value, while execution tainted by 'MyAddon')
```

WoW 12.0 introduced [secret values](https://warcraft.wiki.gg/wiki/Secret_Values): on tainted
execution paths (i.e. inside addon code), many combat-sensitive APIs return values your code
may **hold and pass around, but not inspect**. Any arithmetic (`+ - * / % ^`, unary `-`) on a
secret value throws immediately — typically mid-combat, where you cannot debug.

## Example that breaks in-game

```lua
local pct = UnitHealth("target") / 100          -- SV001
local off = 5 + UnitHealth("target")            -- SV001
local neg = -UnitHealth("player")               -- SV001
```

## How to fix

Store the value first, guard it with `issecretvalue`, and provide a fallback:

```lua
local issecretvalue = issecretvalue or function() return false end  -- pre-12.0 shim (SV011)

local hp = UnitHealth("target")
if issecretvalue(hp) then hp = 0 end   -- or skip the feature gracefully
local pct = hp / 100
```

Or avoid inspecting the number at all — these APIs accept secret values directly:

- `StatusBar:SetValue()` / `StatusBar:SetMinMaxValues()` — health/power bars
- `FontString:SetText()` and `string.format()` — display
- `ColorCurve` / `Duration` objects — thresholds and timers without arithmetic

## Notes

- taintlint reports `error` when the API is documented `SecretReturns = true` (always secret in
  tainted code) and `warning` when it is conditionally secret (`SecretWhen*`).
- The value is also tracked through local variables (`local hp = UnitHealth(u); … hp / max` is
  caught), one severity step lower.
